Issue #456

Dashboard Race Condition: Orchestrator Shows "Exited"

When ao start runs, the dashboard starts before the orchestrator session is created, causing a false "Exited" status.

Problem Statement

The dashboard incorrectly displays the orchestrator as "Exited" immediately after ao start, even though the orchestrator tmux session is actually running fine.

Root Cause Analysis

There is a startup ordering issue in packages/cli/src/commands/start.ts:

Step 1
ao start starts the dashboard (port 3000)
Step 2
Dashboard immediately polls /api/sessions
Step 3 (PROBLEM)
Orchestrator tmux session hasn't been created yet
Step 4 (PROBLEM)
Dashboard finds stale orchestrator metadata from a previous run
Step 5 (PROBLEM)
enrichSessionWithRuntimeState() calls isAlive() on the old tmux handle → returns false
Step 6 (PROBLEM)
Status is set to killed, activity to exited
Step 7
Orchestrator session is created moments later, but dashboard already cached "Exited" state

Relevant Code Locations

packages/cli/src/commands/start.ts lines 256-407 — runStartup() function, startup order
packages/core/src/session-manager.ts lines 835-857 — enrichSessionWithRuntimeState(), marks killed sessions as exited
packages/core/src/session-manager.ts lines 1192-1410 — spawnOrchestrator(), creates the session

Tags

dashboard race-condition startup tmux

Approaches to Fix

Recommended Approach A: Reorder Startup Sequence

Change the startup order to create the orchestrator session before starting the dashboard. The dashboard would then find a live tmux session on its first poll.

Implementation

// Current order (problematic):
1. Start dashboard
2. Start lifecycle worker
3. Create orchestrator session

// New order (fixed):
1. Create orchestrator session
2. Start lifecycle worker
3. Start dashboard
Advantages
  • ✓ Fixes root cause with minimal code change
  • ✓ No added complexity or timeouts
  • ✓ Works reliably every time
  • ✓ No retry logic or polling overhead
  • ✓ Maintains clean separation of concerns
  • ✓ Easy to understand and test
Disadvantages
  • ✗ Slightly slower dashboard startup (waits for orchestrator)
  • ✗ If orchestrator fails to start, dashboard won't open
Alternative Approach B: Grace Period for Startup

Add a brief startup grace period where enrichSessionWithRuntimeState() skips the isAlive check for orchestrator sessions that were created within the last N seconds.

Implementation

// Add a grace period window (e.g., 30 seconds)
const GRACE_PERIOD_MS = 30_000;

function enrichSessionWithRuntimeState(session: Session): void {
  const age = Date.now() - session.createdAt;
  const isInGracePeriod = age < GRACE_PERIOD_MS;

  if (session.type === "orchestrator" && isInGracePeriod) {
    // Skip isAlive() check during grace period
    // Assume session is starting up
    return;
  }

  // Normal isAlive() check...
}
Advantages
  • ✓ Allows parallel startup (dashboard and orchestrator)
  • ✓ Dashboard opens faster
  • ✓ No risk of blocking on orchestrator failure
Disadvantages
  • ✗ Adds magic timeout constant
  • ✗ Masks real failures during grace period
  • ✗ Grace period may be too short or too long
  • ✗ More complex state logic
  • ✗ Could hide bugs in orchestrator startup
Alternative Approach C: Dashboard Retry Logic

If the dashboard sees an orchestrator session as exited, re-check after a short delay before caching the state.

Implementation

// In dashboard /api/sessions endpoint
async function getSessions() {
  const sessions = await sessionManager.list();

  // If orchestrator looks exited, verify again
  for (const session of sessions) {
    if (session.type === "orchestrator" && session.activity === "exited") {
      await sleep(1000); // Wait 1 second
      const freshState = await sessionManager.getRuntimeState(session.id);
      if (freshState?.isAlive) {
        // Override the cached exited state
        session.activity = "working";
      }
    }
  }

  return sessions;
}
Advantages
  • ✓ Self-correcting without requiring perfect ordering
  • ✓ No impact on CLI startup code
  • ✓ Handles edge cases gracefully
Disadvantages
  • ✗ Slower API response (adds sleep delay)
  • ✗ Adds latency to every /api/sessions call
  • ✗ Still may show "Exited" momentarily before retry
  • ✗ Hard to test reliably
  • ✗ Band-aid fix, not root cause
Alternative Approach D: Stale Metadata Cleanup

Clean up stale orchestrator metadata before creating a new session, preventing the dashboard from finding old data.

Implementation

// Before creating orchestrator, clean up old metadata
async function spawnOrchestrator(...) {
  // Remove any stale orchestrator metadata from previous runs
  await metadataStore.delete("orchestrator");

  // Now create the fresh session
  const session = await createSession(...);
  await metadataStore.set("orchestrator", session);
  return session;
}
Advantages
  • ✓ Prevents stale data from being used
  • ✓ Good hygiene practice anyway
  • ✓ Addresses metadata corruption issues
Disadvantages
  • ✗ Still has timing window between cleanup and creation
  • ✗ Dashboard could query during that window
  • ✗ Doesn't fix the race condition root cause
  • ✗ Only helps if metadata is truly stale
Alternative Approach E: Session Initialization Flag

Add an "initializing" state to sessions that the dashboard treats as "pending" rather than "exited".

Implementation

// Add new ActivityState: "initializing"
enum ActivityState {
  initializing = "initializing",  // NEW
  working = "working",
  idle = "idle",
  exited = "exited",
  // ...
}

// When creating orchestrator, set to initializing
async function spawnOrchestrator(...) {
  const session = {
    ...,
    activity: "initializing" as ActivityState
  };
  await metadataStore.set("orchestrator", session);

  // Start the process...
  const tmuxHandle = await startTmux(...);

  // Update to working once started
  session.activity = "working";
  await metadataStore.set("orchestrator", session);
}

// Dashboard treats initializing as pending, not exited
function getAttentionLevel(session: Session): AttentionLevel {
  if (session.activity === "initializing") {
    return "pending";  // Don't show as exited
  }
  // ...
}
Advantages
  • ✓ Accurate state representation
  • ✓ Clear semantic meaning
  • ✓ Dashboard shows appropriate "starting up" state
  • ✓ Useful for debugging startup issues
Disadvantages
  • ✗ Requires changes to type system
  • ✗ Requires dashboard UI updates
  • ✗ More complex state transitions
  • ✗ If process fails during initialization, state unclear

Recommended Solution: Approach A (Reorder Startup)

Why this is the best approach:

  • Addresses the root cause directly (timing issue)
  • Simplest implementation — just move function call order
  • No new complexity, timeouts, or magic constants
  • Works reliably 100% of the time
  • Easy to test and verify
  • Maintains clean architecture without bandaids

Alternative: Combined Approach (A + D)

For maximum robustness, combine Approach A with Approach D (cleanup stale metadata):

  • Reorder startup (A) — fixes the timing race condition
  • Cleanup stale metadata (D) — adds hygiene, prevents any residual issues

This combined approach adds negligible complexity while providing defense-in-depth against related issues.
